# Knowledge Extraction Agent

/distill Phase 3 sub-agent prompt. Extracts atomic knowledge from organized source files.

## Target File
{file_path}

**Read this file first before starting processing.**

## Knowledge Extraction Rules
- 1 file = 1 atomic piece of knowledge
- Filename: English kebab-case reflecting the content (saved in knowledge/notes/ directory)
- **File creation**: Use `rill mkfile` to ensure timestamp accuracy:
  ```bash
  rill mkfile knowledge/notes --slug {kebab-case-name} --type {record|insight|reference} \
    --field "source={target file path}" \
    --field "tags=[tag1, tag2]" \
    --field "mentions=[people/id, orgs/id]" \
    --field "related=[path1, path2]"
  ```
  Then use Edit to append the body to the output path (frontmatter is already generated by `rill mkfile`)
- Frontmatter schema (`rill mkfile` auto-sets `created`. Specify the following via `--type` and `--field`):
  - type: Required (record = facts/data / insight = observations/interpretations / reference = external citations)
  - source: Required (target file path. Use the exact path from "Target File" above)
  - tags: Optional (max 3, **topics only**. Do not put entity IDs in tags; use mentions instead. Refer to tag desc to select matching tags. Inline array)
  - mentions: Optional (Match entity list against the body text, extract referenced people/, orgs/, projects/ IDs. Format: `mentions: [people/person-id, orgs/org-id, projects/project-id]`. Type prefix required)
  - related: Optional (up to 5 related files discovered during Evergreen check below)
- Body: Follow the language rules specified in CLAUDE.md; technical terms in English
- Add `# Title` at the beginning of the body (a concise title describing the content)
- Use standard Markdown links (no Wikilinks)

### Evergreen Check (Glob/Grep exploration)
**Must be performed before creating any new knowledge.**
1. Extract 3-5 search terms from the key concepts of the knowledge to extract
2. Glob(`knowledge/notes/*{keyword}*`) for candidate search (1-2 times)
3. Grep(`{concept}`, path=`knowledge/notes/`, output_mode=`files_with_matches`, head_limit=10) for content search (1 time)
4. **Record hit/miss status for each search term** (e.g., term A=hit, term B=miss, term C=miss)
5. If a file on the same topic is found:
   a. Read only the **frontmatter** of the found file (max 10 lines. Check `type` and `tags`)
   b. If type differs (e.g., existing is insight, new is record/reference) → **Create new** (specify existing in `related`)
   c. If type is the same and content covers the same topic → **Skip** and report
6. **Partial extraction judgment**: If some search terms missed, the information corresponding to missed terms may not be covered by existing knowledge. Consider creating new knowledge for that portion
7. No matches found → Create new
8. Related files found → Specify in `related`

## Key Fact Accumulation (people/)
For people mentioned in the source, determine if there is new information to add to knowledge/people/{id}.md key facts.
- Target: Only people listed in the shared context People mapping
- Read the target people/ file to check existing key facts
- Do not add semantically duplicate information
- If adding, use Edit to append to the key facts section
- Key fact guideline limit: 20 items

## Key Fact Accumulation (projects/)
For projects mentioned in the source, determine if there is new information to add to knowledge/projects/{id}.md key facts or Competitors.
- Target: Only projects listed in the shared context Projects mapping
- Read the target projects/ file to check existing content
- Key facts: Project progress, important decisions, numerical results
- Competitors: New competitive info (new services, pricing changes, partnerships, exits)
- Watch Keywords: New keywords to monitor
- Do not add semantically duplicate information
- If adding, use Edit to append to the relevant section
- Key fact guideline limit: 20 items

## Shared Context
The following data is injected from the orchestrator's prompt (not included in this file):
- **Tag vocabulary**: YAML list format (name + desc). Refer to desc when selecting tags
- **People mapping**: id → name | aliases | company in extended one-line format
- **Orgs mapping**: id → name (aliases) in one-line format
- **Projects mapping**: id → name (stage, tags) in one-line format

## Read Budget
- Target file: Full Read — 1 time
- knowledge/notes/ existing files: **Frontmatter only** (up to first 10 lines for type/tags comparison during Evergreen check. Full Read prohibited)
- knowledge/people/, knowledge/projects/: Read only for key fact targets (max 3 files)

## Output
After processing, report the following **concisely** (do not return file contents):
- List of created files (paths only)
- Skipped knowledge (existing filename + reason in one line)
- Updated people/ files (additions in one line)
- Updated projects/ files (additions in one line)
- New tag names if used
